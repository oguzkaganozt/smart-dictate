## Purpose

OpenADKit is primarily a modular deployment and integration platform for Autoware, not the default environment for deep Autoware source development or the simplest quick-start demo path. Source-built Autoware remains the right choice for broad package development, upstream contribution, and low-level debugging, while monolithic Autoware containers remain the right choice for simple full-stack evaluation. OpenADKit should win where deployment reality matters: running Autoware as modular vehicle software with service separation, component placement, partial restart, compatibility sets, readiness, observability, release trust, and platform integration across environments such as AutoSD, Podman/systemd, BlueChi, Ankaios, SOAFEE, Eclipse SDV, and future S-Core reference workloads.

## Communication

OpenADKit containers should communicate by default through native ROS 2 DDS over host networking, because this preserves Autoware's existing communication model and is the least intrusive way to run Autoware modules in separate containers without requiring Autoware package changes or a custom internal messaging layer. This host-network DDS mode is the default local compatibility profile, while the near-term SDV-native profiles planned for integration are explicit DDS configuration profiles for predictable discovery and QoS, Zenoh bridge profiles for edge/cloud and multi-host communication, BlueChi-managed Podman/systemd profiles for AutoSD-style service orchestration, Ankaios workload profiles for SDV orchestration, and documented S-Core reference workload experiments that align OpenADKit with future platform services without replacing the ROS 2 DDS baseline.
